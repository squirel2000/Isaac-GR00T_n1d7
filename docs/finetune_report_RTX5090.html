6 · Recommendations

  • Data, not steps, is the bottleneck. Loss plateaus by ~step 1000 on 5 episodes; collect dozens–hundreds of demonstrations to get generalization rather than memorization.
  • Headroom exists (peak ~25/32 GB): raise per-device batch, or enable gradient checkpointing (GRAD_CKPT=1).
  • Closer-to-default optimizer: uv pip install bitsandbytes then OPTIM=paged_adamw_8bit (verified on sm_120 — see §7-B).
  • Stability: keep the GPU clock/power cap (§7-E) installed for any long run.

7 · Appendix — operating notes (new dataset, 8-bit Adam, monitoring, stability)

A · Fine-tuning a new NEW_EMBODIMENT dataset → --modality-config-path

Two distinct files: the dataset's meta/modality.json (splits the flat state/action vectors into named field groups — usually shipped) and the Python config you author that registers a ModalityConfig (which keys, prediction horizon, action representation). Write one ModalityConfig per block with modality_keys matching modality.json exactly, one ActionConfig per action key, then register_modality_config(cfg, EmbodimentTag.NEW_EMBODIMENT).

Worked example (validated, 417 shards / 426 315 samples, no key errors): bimanual OpenArm-O6 CanSorting — 26-D state/action (left_arm[0:7] right_arm[7:14] left_hand[14:20] right_hand[20:26]), 1 RGB cam camera, arms RELATIVE + hands ABSOLUTE, horizon 16 → examples/OpenArm_O6/openarm_o6_config.py.

B · 8-bit Adam (paged_adamw_8bit)

Verified on this sm_120 GPU (bitsandbytes 0.49.2). Optimizer state ~3 GB (vs 13 GB fp32 AdamW / ~10 MB adafactor); 50k config peaks ~27.5 GB / 32 GB. Enable with uv pip install bitsandbytes then export OPTIM=paged_adamw_8bit (unset / OPTIM=adafactor to use adafactor).

C · Monitoring & Weights & Biases

No wandb: tail -f <log> (tqdm bar), watch -n2 nvidia-smi, and read log_history in the newest checkpoint-N/trainer_state.json for live loss/grad-norm/lr (stdout loss lines are buffered to the file).
wandb: uv run wandb login once, then add --use-wandb [--wandb-project NAME] to the launch command; open the run URL printed at startup. Offline box: WANDB_MODE=offline + later wandb sync.

D · Rank 0, Worker 0: Caching shard...

From GR00T's streaming sharded dataset, not the optimizer. Episodes are grouped into shards (~1024 frames each); the first time the sampler touches an uncached shard it decodes its videos and caches it (Wait for shard N ... 8.24s, then 0.00s on later hits). With --dataloader-num-workers 0 it briefly blocks the main process → minor, normal overhead. Independent of paged_adamw_8bit.

E · Spontaneous reboots under sustained load — RTX 5090 power

Repeated hard reboots minutes–tens-of-minutes into long runs. last shows the session ends in crash; the kernel log just stops (no panic / Xid / thermal / MCE), power back ~80 s later. That is an instant power loss — not software/CUDA/GR00T — the PSU tripping on the 5090's transient spikes. Escalation observed: no cap → reboot @1700–2500 steps; -lgc 1800,2200 -pl 400 → reboot @~5754; so the clock lock is the key lever (a power cap still allows boost spikes; locking max clock holds voltage/current down).

Fix (installed & verified here, persistent via systemd): nvidia-smi -pm 1; nvidia-smi -lgc 510,1500; nvidia-smi -pl 400 → under load SM clock pins at ~1275 MHz, ~290 W (vs 500–600 W+ unlocked). Made permanent as nv-powercap.service so it re-applies every boot (runtime nvidia-smi settings otherwise reset on reboot). Full apply/remove commands in README §9.
If it still reboots at 1500/400 W → hardware: PSU here is 1200 W (enough watts), so reseat the 12V-2x6 connector (prime suspect — a previously-working cap started failing again) and check the PSU (ATX 3.1, single high-current rail). Every run auto-resumes from the last checkpoint.
